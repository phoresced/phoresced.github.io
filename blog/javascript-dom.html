Remember HTML and CSS?

Okay, a refresher.
HTML and CSS are both languages which make things happen on a website, and they're essentially complementary opposites.

HTML is like the skeleton, the bones in your body.
CSS is the external elements of your body; your skin, eye and hair colour, hair length, clothes, makeup (if you want it).
You do need some of those things, while others are purely decorative. If I wanted to whisper in your ear, I need to locate your ear, and position myself in relation to it.
The mechanism of the ear needs to be linked to the external representation - just like the HTML and the CSS need to be linked so someone can navigate a website.

Does that make sense so far?
Cool.

So how does Javascript fit into that?

Imagine all the other systems and processes contained in your body which make it function. We introduce these functions by using Javascript.

Whether it's white blood cells pumped by the heart, or food being accessed for energy, these functions are interactive - they listen for changes in your state and respond with the appropriate elements and measures.

If your skeleton doesn't have a space to carry an area of your body, then you can't write a function for it. So we have to link the javascript back to the HTML.

As a human interacting with this body, I still need to navigate the webpage in order to cause things to happen, so I need CSS to tell me where your ear is.

But even if my noises rattled your ear bones, the sound would have no meaning without a brain function who could pass those noises into your sensemaking center. We use Javascript functions to make sense of the interactions being performed on and with the website.

Control flow and loops in Javascript

Now imagine you are cooking dinner. Whether you're following an existing recipe or improvising, you'll want to sequence the steps so every ingredient cooks and infuses at the right time.

In javascript, we have a few tools to ensure things happen at the right time.

We set a flow running with events, rather than making every possible action happen as soon as a page is loaded. This is important for guiding interactions. Maybe the vistor on my page needs to click a button to make something happen. Unless they click the button, I don't have permission to trigger the response.
So I code something to watch for an action or state, like "onions have softened", and then a flow is triggered, "add garlic". If that ingredient was added at the same time as the onions, it would burn, because it takes less time to cook.


Another tool we can use is loops.
The loop recurs as many times as it needs to until something stops it. Kinda like your culinarily-ignorant friend helping you stir the onions so they don't burn. You give them one instruction: stir. The loop follows a motion on the webpage within the conditions that we set. It might listen for new form submissions, or clicks, or enter keys; and it can count the total amount of times that somethings happens, because unlike your hungry friend, the loop is an algorithm who loves maths. If we tell it to count, it's not going to lose count. It just keeps stirring the pan and logging stirs.

This tool can be helpful in a lot of ways when we're controlling the flow of functions on our webpage. We don't need to write new functions every time we want to pan to be stirred, we just tell our algorithm friend to repeat until, let's say, the tomatoes are added plus five more stirs (to get that tomato nicely mixed in there.

The DOM (Document Object Model)

Javascript knows how to interact with the internet because it has a model of the structure like blueprints to the bank vault. It knows where all the keys are hidden and which door goes where.

Every website is written in a document, usually of the HTML kind, and we know how they work.

Compared to human anatomy, these documents can grow virtually infinite limbs, called Objects. So whenever a new Object is made in the document, the model follows this and the related blueprint information is copied over.


Oh, you want to grow an image? I know how images behave. Let me load up some possible things that an image might do.

We can follow the model to inspect the code on any website and figure out what's going on, as well as playing with how we might change the objects, slightly or dramatically.

I might want to see what would happen if I put more padding in each paragraph. Or I might look into what processes are triggered automatically when I press the red button and make a more conscious decision about clicking random buttons on strangers' websites.

And we don't have to be on a site to play with the DOM - we can just load a javascript file and manipulate the model in our code editing app. If you tell the model to do something, then push those new instructions to the original HTML file (assuming you have permission to do that), then the changes will occur.
So even though it's called a model, it's not a model as muggles know them to be.

Arrays and Objects

So, we want to make something happen to our meal. We've written instructions so that our wizard robot friend could cook it for us while we're driving home. How does it access the data to follow?

We might use arrays when our instructions relate to a list; like a collection of things or possible choices. When the instructions relate to individual things which potentially have no relationship, they would each need to be written as an object.

An array can be inserted as a category (aka key) nested in the object as one of its pockets of data; likewise the items listed in the array can be given properties if we wrote them as objects. Let's go back to our stove and figure out what that means for our meal.

-----

We're making a curry. The onions, garlic and tomatoes are already simmering away, great, that's the basics taken care of. But what else should go in?

We could use an array to list all the vegetables we like. We'd like to reference this list and select two of the items in the list, at random perhaps.

But do we have any in the fridge? How about the pantry? The freezer?

The array lists cauliflower, potato, peas, carrot, okra.

Each of these will also be defined in javascript as an object. We want to keep stock of whether each vegetable is available and where it's stored. So we might write this object:

let cauliflower = { pantry: 0, fridge: 1, freezer: 0 }

Pantry, fridge and freezer are keys and while maybe we can't rob a bank with these keys, we can certainly learn where our friend should look in the house for ingredients.

And if I brought more carrots home, I could log them when I get home by referencing the object and then the key and telling it to count +1 for each carrot I restocked.


We might write a condition in the selectVeges function which ignores any vegetable in the array whose total stock count is 0.

So our wizard robot friend can choose any two of the items in the array whose object properties are greater than or equal to 1. But how do we get the data from an array?


Arrays will count incrementally (so, the usual way of counting - a sequence where the next item counted is +1 from the last item), and it starts its count at 0.

To get the data about carrots, we query item 3, because cauliflower is 0, potato is 1, peas are 2, and the next increment is 3. We listed carrots after peas and before okra. This matters as the position assigns the number.

So using our array name and counting the stock of every "position in the array", we learn that veges[0]=1, veges[1]=5, veges[2]=1 and so on. We have one cauliflower, five potatoes, 1 bag of peas, etc.


Our friend selects the potatoes and cauliflower, and returns them from the pantry and the fridge, respectively.

Functions

Now that our meal's sorted, we can return to our original comparison of HTML, CSS and Javascript, and the analogy of the bodily functions being written in the language of JS.

To be more specific about functions, let's think about how many different things can be going on in even a simple interactive website these days.

We live in the age of social media, where we are constantly reacting to posts, bookmarking things for later, making connections, sharing things... the interactivity is what we love about the internet.

Because these are recurring actions, we want to write code which recognizes that action no matter when or how often it might occur.

We give the function a name, some round brackets, and a whole lot of statements inside some curly brackets. These statements know what to check for, and can get really complicated, like giving a different output to different people depending on what it knows about them (like, are they logged in? Do they have adminstrative permissions? and whatever other data you feel comfortable sharing with potentially anonymous web developers).

The function gets triggered whenever it gets called elsewhere in the script. Maybe we simply write its name and put some round brackets there, (), like opening a portal.

Or, we don't want the webpage to just rush ahead and perform the function, so we associate it to an event like 'onclick' or 'keyup' (i.e. our website visitor clicks the related button, or finishes pressing a button.)

That's it! Duplicate the function immediately by calling it in the context of whatever else is going on.

It saves a whole lot of time and work because the code only has to refer to the original function in order to process an action, and we can pack a lot of stuff in between those curly brackets.

Summary

Javascript has a lot of fun tools that brings a simple skinned skeleton alive with living, breathing processes. We anticipate humans to come along and animate the space, and are able to make things happen anew for every vistor, and in a relationship with them that can build over time, as data references other data, triggers functions, and gives the code a purpose for being.

So what kind of portals do we want to open up? What can this new life form do with its existence? A lot of good, bad and chaotic neutral outcomes are possible with this magic wizard language.
